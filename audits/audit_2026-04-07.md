# Audit Report: April 7, 2026

**Audited Modules:** `internal/patient`, `internal/staff`, `internal/clinic`, `sal_app` (design system)
**Project-wide:** Security, Multi-tenancy, Testing Strategy

## Final Assessment

**Grade: B**

The project continues to demonstrate high-quality engineering, particularly in its approach to PII encryption and API design. The most significant achievement since the last audit is the introduction of a robust testing suite for the `staff`, `clinic`, and `patient` modules, filling a critical gap identified on April 5. However, the grade is tempered by a newly discovered **critical security vulnerability** in the patient listing logic and a growing contradiction between the project's foundational plan and its current engineering rules regarding data access.

**Key Improvements:**
1.  **Patient Module Delivery:** The `internal/patient` module is feature-complete, implementing complex multi-tenant logic and transparent PII encryption/decryption.
2.  **Test Coverage Expansion:** Comprehensive unit and integration tests now cover all primary domain modules (`auth`, `staff`, `clinic`, `patient`), significantly reducing regression risk.
3.  **Stability:** Critical fixes from previous audits (refresh token rotations and clinic update authorization) remain stable and verified.

**New/Remaining Issues:**
1.  **CRITICAL BUG (Authorization Bypass):** In `internal/patient`, the `ListSubjects` endpoint returns **all clinic patients** if a staff member has neither `view_all_patients` nor `view_own_patients` permission. The logic defaults to no filter rather than an empty set.
2.  **SQLC Conflict:** There is a direct contradiction between `BACKEND_PLAN.md` (which mandates `sqlc`) and `CLAUDE.md` (which now explicitly forbids it). All `sqlc` traces have been removed, favoring handwritten SQL. This needs a definitive decision to avoid future "refactor back-and-forth."
3.  **Permissive Staff Access:** The `/api/v1/staff` endpoints allow any authenticated staff member to view full profiles and permission sets for all other staff, which may be too permissive for larger clinics.

---
## Module Audit: `internal/patient`

**Grade: C (due to security flaw)**

*   **Status:** Excellent implementation of vertical-specific logic (Veterinary details) and PII protection.
*   **Vulnerability:** `ListSubjects` fails safe. If `OwnerScope` is false (calculated as `!ViewAll && ViewOwn`), the repository returns the full clinic list.
*   **Recommendation:** Update `ListSubjects` in the service or repository to return an empty list or `ErrForbidden` if the caller has no view permissions.

---
## Module Audit: `internal/staff` & `internal/clinic`

**Grade: A-**

*   **Status:** Both modules are now "fully audited" with the addition of comprehensive test suites.
*   **Recommendation:** Consider restricting the `list-staff` and `get-staff-member` endpoints to users with `manage_staff` or at least masking sensitive permission fields for non-admins.
